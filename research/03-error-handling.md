# Error Handling — Current Consensus

Rust splits failure into two worlds: **recoverable** errors, modeled with `Result<T, E>`, and **unrecoverable** errors, modeled with `panic!` [1]. This is not a stylistic choice — it is the type-level contract that the rest of the ecosystem is built on. As of stable Rust ~1.85+ and the **2024 edition** [2], the patterns below have converged into a stable, widely-agreed consensus, with `thiserror` 2.x and `anyhow` (both by David Tolnay) as the de facto crates [3][4]. This report is opinionated: each section gives the good practice, the mistake to avoid, and why.

## 1. `Result` and the `?` Operator Are the Default

`Result<T, E>` is the baseline. The `?` operator unwraps an `Ok`/`Some` and **early-returns** the `Err`/`None`, replacing the old `match` boilerplate [5]. The non-obvious part: on the error path, `?` calls `From::from` on the error, converting it into the function's declared error type [5]. This single fact is the mechanism behind nearly every ergonomic error pattern in Rust — `From` impls (often generated by `thiserror`'s `#[from]`) are what make `?` "just work" across heterogeneous error sources.

```rust
fn load(path: &Path) -> Result<Config, MyError> {
    let text = fs::read_to_string(path)?; // io::Error -> MyError via From
    let cfg  = toml::from_str(&text)?;    // toml::Error -> MyError via From
    Ok(cfg)
}
```

**Good practice:** Make fallible functions return `Result` and propagate with `?`; design your error type so the needed `From` conversions exist [5].

**Mistake to avoid:** Reaching for `.unwrap()`/`.expect()` to silence the compiler in real code paths. Each is a latent `panic!`. Use them only where a failure is genuinely a *bug*, and prefer `expect("reason")` over `unwrap()` so the panic message documents the invariant you are relying on [1].

## 2. When to `panic!` vs. Return `Result`

The official Book gives a crisp rule: **return `Result` by default; `panic!` only when the code reaches an unrecoverable bad state** with no reasonable recovery for the caller [1].

Return `Result` when **failure is an expected possibility** — malformed parser input, an HTTP rate-limit response, a missing file. These are not bugs; the caller must decide what to do [1].

`panic!` is appropriate when:

- **A contract/invariant is violated.** A function whose behavior is only guaranteed for certain inputs may panic when those inputs are wrong, because a contract violation is *always a caller-side bug* the programmer must fix, not handle at runtime [1]. Document such panics in the API docs [1][6].
- **Continuing would be insecure or harmful** (e.g., out-of-bounds access) [1].
- **Examples, prototypes, and tests** — `unwrap`/`expect` keep them uncluttered, and in tests a panic *is* the failure signal [1].

Effective Rust frames the working default even more aggressively for libraries: **"Don't panic"** — prefer returning errors, because a panic in a library forces an abort/unwind decision onto every downstream user [6].

**Good practice:** Encode invariants in the type system so bad states are unrepresentable (e.g., a `Guess` newtype that validates `1..=100` in its constructor), rather than re-checking everywhere [1].

**Mistake to avoid:** Using `panic!`/`unwrap` for *expected* runtime conditions like bad user input or network failure. That converts a recoverable situation into a crash and removes the caller's agency [1].

## 3. The `std::error::Error` Trait Is the Interop Contract

Every well-behaved error type should implement `std::error::Error` [7]. The trait is small and stable:

```rust
pub trait Error: Debug + Display {
    fn source(&self) -> Option<&(dyn Error + 'static)> { None }
    // description() and cause() are DEPRECATED — use Display and source()
}
```

Key points from the std docs [8]:

- It **requires `Debug` + `Display`**. `Display` is the human-facing message; `Debug` is diagnostic.
- `source()` exposes the **underlying cause**, forming a navigable error chain across abstraction boundaries [8].
- **Critical rule:** expose a wrapped cause via *either* `source()` *or* your `Display` output — **not both** — or chain printers will duplicate it [8].
- `description()` (deprecated 1.42) and `cause()` (deprecated 1.33) are dead; do not implement them [7][8].
- `provide()` (the type-based context API, e.g. for `Backtrace`) remains **nightly/experimental** [8].

The API Guidelines (`C-GOOD-ERR`) add the interop requirements [7]:

- Implement `Send + Sync + 'static` too. Non-`Send` errors can't cross `thread::spawn`; non-`Sync` can't be shared via `Arc`; and `'static` is what enables `downcast_ref`. `Error + Send + Sync + 'static` is the maximally useful trait-object bound [7].
- `Display` messages should be **lowercase, no trailing punctuation** ("invalid IP address syntax", "unexpected end of file") [7].

**Mistake to avoid:** Using `()` as an error type. It has no `Error`/`Display` impl, breaks `?`, and gives useless output. Even a unit struct is better [7].

## 4. `thiserror` for Libraries, `anyhow` for Apps — and the Better Framing

The popular heuristic — **`thiserror` for libraries, `anyhow` for applications** — is correct often enough to be the standard advice [3][9]:

- **`thiserror`** derives `Error`/`Display`/`From` on your own enums/structs with minimal boilerplate, giving callers a concrete type to **match on** [3]. Its `#[from]` generates the `From` impls that power `?`, and a `Backtrace`-typed field is surfaced via the `provide()` mechanism [3].
- **`anyhow`** provides a single opaque `anyhow::Error` (a smart `Box<dyn Error + Send + Sync>`) plus a `Context` trait whose `.context("...")` enriches errors as they propagate [4][9]. Ideal where you just need to report or log, not branch on the failure [4].

```rust
// Library: a typed enum (thiserror)
#[derive(thiserror::Error, Debug)]
pub enum StoreError {
    #[error("record {0} not found")]
    NotFound(u64),
    #[error(transparent)]
    Io(#[from] std::io::Error),
}

// Application: opaque + context (anyhow)
fn run() -> anyhow::Result<()> {
    let cfg = load_config().context("failed to load config")?;
    Ok(())
}
```

**The deeper consensus (Luca Palmieri):** the library-vs-app split is a *proxy* for the real question — **intent** [9]. Ask: *does the caller need to behave differently based on the failure mode?*

- **Yes → use an error enum** (with `thiserror`) so callers can pattern-match. Libraries usually answer "yes" because the author can't predict the consumer's needs — which is *why* the heuristic mostly holds [9].
- **No → use an opaque error** (`anyhow::Error` or `Box<dyn Error>`), even inside a library [9].

Palmieri's other practical rules [9]:

- **Avoid "ball of mud" enums.** One variant per internal call site is a code smell that leaks implementation detail. Collapse toward what the caller can act on (e.g., `ValidationError` vs `UnexpectedError`).
- **Errors are both control flow and reporting.** Internally: enum variants / HTTP status codes for branching; logs and response bodies for humans.
- **Log errors only when they are *handled*** — typically at the boundary/middleware — never on every `?` hop, or you get duplicate noise [9].

**Mistake to avoid:** Exposing `anyhow::Error` in a *library's public API* when consumers might need to branch — you've thrown away the type information they need. Conversely, hand-rolling a giant enum in an app's `main.rs` where nobody matches on it is pointless boilerplate; `anyhow` is the right tool there [3][9].

## 5. `Box<dyn Error>` — the Std-Only Middle Ground

Before adding dependencies, `Box<dyn std::error::Error>` is the standard-library way to unify heterogeneous errors [10]. Any `T: Error` auto-converts into it via a blanket `From`, so `?` works seamlessly, and a type alias keeps signatures clean:

```rust
type Result<T> = std::result::Result<T, Box<dyn std::error::Error>>;
```

It is the right call for quick prototypes, one-off `main`, or glue code where you don't need to match on variants [10]. `anyhow::Error` is essentially a more ergonomic, context-carrying superset of this pattern (with a guaranteed backtrace and `Send + Sync`) [4][9].

**Good practice:** Use `Box<dyn Error + Send + Sync + 'static>` if the value may cross threads; the `'static` bound also re-enables `downcast_ref` to recover the concrete type when you must [7][8].

**Mistake to avoid:** Putting `Box<dyn Error>` in a library's public signature when callers need to discriminate failures — downcasting by concrete type is brittle and undocumented compared to a real enum [9][10].

## 6. Error Context: Add Meaning as You Propagate

A bare `io::Error` that says "no such file" without saying *which* file is nearly useless in production. The consensus is to **attach context at each boundary** [9].

- With `anyhow`: `.context("while reading user config")` / `.with_context(|| format!("path: {}", p.display()))` [4][9].
- With `thiserror`: carry the context in your variant fields (`#[error("failed to read {path}")] ReadConfig { path: PathBuf, source: io::Error }`) and chain via `source` [8][9].

Implementing `source()` correctly lets a generic walker print the full causal chain — strictly better than logging one level [9].

**Mistake to avoid:** Swallowing the underlying error when adding context (e.g., discarding the `source`). You lose the root cause, which is usually the most actionable part of the chain [8][9].

## 7. 2024-Edition Notes

The 2024 edition (stable since Rust 1.85, Feb 2025) doesn't overhaul error handling, but one relevant change: the **never type (`!`) fallback** is now `!` instead of `()`, and `never_type_fallback_flowing_into_unsafe` is deny-by-default [2][11]. This makes `Result<T, !>` and diverging expressions behave more intuitively in error-adjacent code; rarely, code that relied on the old `()` fallback needs a fix during migration [11]. The richer **`Error::provide` / `Request`** context API remains nightly, so portable backtrace propagation still flows through `anyhow`/`thiserror` today [8].

## Summary Cheat-Sheet

- Default to `Result` + `?`; reserve `panic!`/`unwrap`/`expect` for bugs and contract violations [1][6].
- Implement `Error: Debug + Display` (+ `Send + Sync + 'static`); never use `()` [7][8].
- Library, callers branch on failure → typed enum via `thiserror` [3][9].
- App or "caller just gives up" → opaque `anyhow::Error` with `.context()` (or std `Box<dyn Error>`) [4][9][10].
- Choose by **intent**, not by the library/app label [9].
- Add context at boundaries; preserve `source`; log only where handled [8][9].
